# Everything Claude Code 深度使用实战手册（Windows 版）

## 1. 这份手册解决什么问题
你已经完成了 ECC 的大规模安装，但真正拉开效率差距的不是“装了多少”，而是“是否形成稳定可复用的工作流”。
本手册的目标是：
- 让你把 ECC 从“工具集合”变成“可复制的产出系统”
- 让你在复杂任务中稳定拿到高质量结果，而不是靠运气
- 让你知道什么时候该用技能、什么时候该用子代理、什么时候该直接执行

---

## 2. 先建立正确心智模型
ECC 可以理解为四层能力：
1. 规则层（rules）
   - 约束编码风格、测试要求、安全底线、提交流程
2. 技能层（skills）
   - 给出某一领域的成熟做法，比如测试、部署、前端、后端、研究
3. 代理层（agents）
   - 把专业分工显式化，例如 planner、tdd-guide、code-reviewer、security-reviewer
4. 执行层（你的命令与终端）
   - 真正落地安装、改代码、跑测试、部署发布

结论：
- rules 决定下限
- skills 提高均值
- agents 决定复杂任务上限

---

## 3. 你的当前环境建议基线（Windows）
建议固定以下三件事，避免后续不稳定：
- PowerShell 7 为主终端
- Node.js LTS（建议 20+）
- Git 全局代理和证书策略正确

快速自检命令：
```powershell
node -v
git --version
npx skills list | Measure-Object
```

如果你经常遇到网络抖动，建议准备两条策略：
- 重试策略：每个安装命令最多重试 3 次
- 回退策略：单个 skill 失败不阻塞整体批量执行

## 4. 标准作业流（建议直接照抄）

### 阶段 A：需求澄清（避免返工）
每次开始前，先把需求写成三行：
- 目标结果：最终可交付物是什么
- 约束条件：语言、框架、目录、时间、风险边界
- 验收标准：怎么判断“完成”

你可以直接用这段模板：
```text
目标：
约束：
验收：
```

### 阶段 B：先规划，再编码
复杂任务先调用 planner 或架构思考，不要直接开写。
适用场景：
- 涉及 3 个以上文件改动
- 涉及数据库/API/权限模型
- 涉及跨端（前端+后端）联动

### 阶段 C：测试前置（TDD）
修 bug 或加功能时，优先让 tdd-guide 先给失败测试路径。
你至少要保证：
- 新功能有对应测试
- 回归路径可复现
- 改动后能自动验证

### 阶段 D：强制代码评审
代码改完后，默认再过一轮 code-reviewer。
如果改动涉及输入处理、鉴权、文件上传、密钥，追加 security-reviewer。

### 阶段 E：交付与沉淀
交付时输出三类内容：
- 改了什么
- 怎么验证
- 风险和后续建议

并把可复用经验沉淀成模板，放进你的个人笔记或项目规范中。

## 5. 高频实战场景（可直接复用）

### 场景 1：前端美化 + 体验升级
建议组合：
- frontend-patterns
- vercel-react-best-practices
- web-design-guidelines（做设计审查时）

执行顺序：
1. 先让代理给出视觉方向与组件改造计划
2. 先做关键页面（首页/落地页/核心流程页）
3. 再做全局样式和性能优化
4. 最后做无障碍和移动端检查

验收重点：
- 首屏性能
- 交互动效是否“有意义”而非噪音
- 文案与视觉是否同一语气

### 场景 2：全栈功能开发（最常见）
建议组合：
- backend-patterns / api-design
- frontend-patterns
- tdd-workflow
- verification-loop

执行顺序：
1. 先定义 API 合同（输入、输出、错误码）
2. 后端测试先写，前端按合同联调
3. 全链路回归（成功流、失败流、边界流）

### 场景 3：发布上线
建议组合：
- deployment-patterns
- deploy-to-vercel 或 vercel-cli-with-tokens
- security-review

上线前清单：
- 环境变量齐全且无硬编码密钥
- 健康检查可用
- 回滚路径明确
- 关键日志可追踪

## 6. 失败率最高的 10 个坑（重点）
1. 一上来就写代码，不先收敛需求
2. 安装很多技能，但没有固定执行顺序
3. 没有把“测试通过”设成硬门槛
4. 只做功能验证，不做异常路径验证
5. 忽略安全检查（尤其输入和密钥）
6. 失败后直接换方案，不做最小复盘
7. 命令一次性跑太大，失败后难定位
8. 网络抖动时无重试机制
9. 交付只给结果，不给验证方法
10. 不沉淀模板，重复踩同一个坑

## 7. 稳定性与排障手册

### 7.1 安装类问题
常见现象：clone 失败、npm 403、模块缺失。
处理顺序：
1. 先确认网络和代理
2. 再确认 Node/npm 版本
3. 失败命令单独重试 3 次
4. 仍失败就降级为逐个安装，不要全量并发

### 7.2 调用类问题
常见现象：技能已安装，但似乎没有生效。
处理顺序：
1. 确认技能目录和 SKILL.md 在位
2. 提示词中明确触发场景与目标
3. 把大任务拆小，减少上下文污染
4. 对关键步骤给出可验证输出要求

### 7.3 质量类问题
常见现象：能运行，但质量不稳定。
处理顺序：
1. 增加测试覆盖和回归集
2. 强制加入 code-reviewer
3. 涉及安全边界时加入 security-reviewer
4. 为重复任务建立固定模板

## 8. 建议你固定的“日常操作模板”

### 模板 1：新功能开发
```text
请按以下流程执行：
1) 先做需求澄清（目标/约束/验收）
2) 给出最小实现计划
3) 先写测试再实现
4) 实现后做代码评审
5) 输出验证结果与风险清单
```

### 模板 2：线上问题修复
```text
请先复现问题并定位根因，不要先改代码。
然后按“最小改动+测试补齐+回归验证”执行。
最后输出：根因、修复点、验证证据、是否有同类风险。
```

### 模板 3：前端优化
```text
目标：提升视觉质量与核心指标（首屏、交互、可访问性）。
请先给改造优先级，再逐步实现并量化前后差异。
```

## 9. 维护策略（周/月节奏）
- 每周：清理低频技能，保留高价值组合
- 每两周：更新一次规则和关键技能
- 每月：复盘 3 次高价值任务，提炼可复用模板

你会发现：
真正提升效率的不是“装更多”，而是“固定高成功率流程”。

---

## 10. 最后的执行建议
从今天开始，把每次任务都走这条主线：
- 先定义验收，再动手
- 先计划分解，再实现
- 先测试兜底，再交付
- 先复盘沉淀，再进入下一个任务

只要连续执行两周，你的稳定性和交付速度会明显提升。
